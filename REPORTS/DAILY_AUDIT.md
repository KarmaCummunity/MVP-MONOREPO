# 🛡️ Daily Audit Report: May 4, 2026

## 🚨 Critical Vulnerabilities
 * Hardcoded credential material is committed in repository files, including a production-style JWT secret in `docs/SSOT/runbooks/api/environment-separation.md:125-126`, Redis password material in `docs/SSOT/runbooks/api/redis-status-summary.md:15` and `docs/SSOT/runbooks/api/package-readme-archive.md:34`, a development JWT secret in `apps/api/railway.toml.dev:33-34`, and an environment dump at `apps/api/railway_vars.txt`. Production-like backup JSON files are also present under `apps/api/data-backups/20251224-162421/`. - File: docs/SSOT/runbooks/api/environment-separation.md; docs/SSOT/runbooks/api/redis-status-summary.md; docs/SSOT/runbooks/api/package-readme-archive.md; apps/api/railway.toml.dev; apps/api/railway_vars.txt; apps/api/data-backups/20251224-162421/*.json -> Action: Rotate exposed JWT/Redis/Google/DB credentials, remove secrets and backups from the repository, add ignore/secret-scanning controls, and audit git history for prior exposure.
 * Sync endpoints fail open when `SYNC_API_KEY` is unset because the runtime default is `change-me-in-production` and the API-key check is skipped for that value. An authenticated admin can invoke Firebase sync without the intended second factor. - File: apps/api/src/controllers/sync.controller.ts:26-28, apps/api/src/controllers/sync.controller.ts:416-424 -> Action: Fail closed in non-development environments when `SYNC_API_KEY` is absent, require a strong `X-API-Key` for sync routes, and remove the placeholder as a runtime default.
 * Public/demo operational endpoints are registered without authentication and can inspect or mutate rate-limit/session state. `RateLimitController` exposes status, stress-test, stats, and clear operations without guards; `SessionController` is marked demo/test but exposes login, validation, session listing, and logout-all without guards while still registered through `AuthModule`/the backend module graph. - File: apps/api/src/controllers/rate-limit.controller.ts:20-218; apps/api/src/auth/controllers/session.controller.ts:6-8, apps/api/src/auth/controllers/session.controller.ts:136-222 -> Action: Remove demo controllers from deployed builds or protect all diagnostic/session management routes with admin authentication and environment gating.
 * Donation reads and mutations expose privacy and IDOR risks. Public `GET /api/donations/:id` joins and returns `donor_phone` and `donor_email`, while authenticated `PUT` and `DELETE` routes update/delete by donation ID only with no owner or admin check. - File: apps/api/src/controllers/donations.controller.ts:667-695, apps/api/src/controllers/donations.controller.ts:698-779 -> Action: Strip donor contact fields from public reads, require ownership or admin authorization for mutation routes, and add tests for cross-user update/delete denial.
 * Registration/login can grant admin role based on a hardcoded fallback email instead of the configured `ROOT_ADMIN_EMAIL` bootstrap path. This conflicts with the SRS security requirement that admin access is role-based and avoids hardcoded email checks except the configured bootstrap. - File: apps/api/src/config/kc-root-admin.defaults.ts:1-6, apps/api/src/services/user-auth.service.ts:101-102, apps/api/src/services/user-auth.service.ts:212-220; SRS: docs/SSOT/SRS/03-non-functional-requirements.md:28-34 -> Action: Remove admin promotion from normal registration/login, require explicit `ROOT_ADMIN_EMAIL` bootstrap with verified account ownership, and migrate existing roles deliberately.

## 🐞 Bugs & Logic Issues
 * Profile Open tab fetches all rides with `enhancedDB.getRides({})` and filters client-side for the viewed user. In backend mode this can pull unrelated rides into the client, increasing privacy exposure and scaling poorly. - Recommendation: Use `apiService.getUserRides(targetUserId, 'driver')` or add a server-filtered owner endpoint so filtering is enforced before data leaves the API. File: apps/mobile/bottomBarScreens/profile/openRouteLoadOpenContent.ts:166-180; apps/mobile/utils/enhancedDatabaseService.ts:515-528; apps/mobile/utils/apiService.ts:648-650.
 * Some API error paths expose internal verifier details to clients. Firebase token verification failure messages are interpolated into `UnauthorizedException`, and some controllers return raw `error.message` strings. - Recommendation: Log detailed errors server-side and return generic external error messages consistent with CODE_QUALITY constraints. File: apps/api/src/auth/guards/jwt-auth.guard.ts:207-213; apps/api/src/auth/controllers/session.controller.ts:123-128.
 * Environment verification comments include identifiable database password prefixes and host hints, which can aid infrastructure fingerprinting if logs or source are exposed. - Recommendation: Replace password-prefix checks with environment labels, instance IDs stored in secrets, or redacted fingerprints. File: apps/api/src/main.ts:149-178.

## 🏗️ Architecture & Clean Code
 * Controllers still mix HTTP routing, business rules, direct SQL, Redis/cache behavior, and some DDL-style setup despite the CODE_QUALITY target of clear Controller/Service separation and inward dependencies. This is documented as legacy in the SRS for Donations/Rides, but it remains a high-risk area for security-sensitive authorization bugs. - File: apps/api/src/controllers/donations.controller.ts; apps/api/src/controllers/rate-limit.controller.ts; docs/SSOT/SRS/05-backend-architecture.md:66-78 -> Action: When touching these flows, extract authorization/business logic into services/use-cases and add focused unit tests around ownership and privacy boundaries.
 * Mobile profile loading bypasses the central authenticated API client by importing raw `axios` for item delivery calls, which can miss shared auth refresh, retry, and error handling. - File: apps/mobile/bottomBarScreens/profile/openRouteLoadOpenContent.ts:68-89 -> Action: Route these calls through `apiService`/`authFetchInterceptor` and keep API access behind a single application-layer boundary.
 * The WebView donation flow enables JavaScript and DOM storage for a third-party site without origin hardening. - File: apps/mobile/screens/WebViewScreen.tsx:71-80 -> Action: Prefer the system browser for third-party/payment-like flows or configure a strict origin allowlist and navigation guard if WebView remains required.
