## ЁЯЪи Critical Vulnerabilities

* Authenticated profile update IDOR: `PUT /api/users/:id` is protected by `JwtAuthGuard`, but the controller passes the path id directly into `UserProfileService.updateUser` without comparing it to the authenticated session user. The service updates sensitive fields including `password`, `firebase_uid`, `roles`, and `settings` for the resolved user. - File: `apps/api/src/controllers/users.controller.ts:126`, `apps/api/src/services/user-profile.service.ts:254` -> Action: Require the current session user id for self-service updates, allow role/security-field updates only through explicit admin endpoints, and reject body/path identity mismatches.
* Authenticated social action spoofing: post likes use `body.user_id` as the actor even though the route is JWT-protected, allowing an authenticated caller to like/unlike as another user if they know the victim UUID. - File: `apps/api/src/controllers/posts.controller.ts:1343` -> Action: Derive the actor from `@CurrentUser()`/request session and ignore or reject client-supplied actor ids on all like/comment mutation endpoints.
* Donation ownership bypass: `PUT /api/donations/:id` and `DELETE /api/donations/:id` only filter by donation id, not by `donor_id` or admin authorization. - File: `apps/api/src/controllers/donations.controller.ts:698`, `apps/api/src/controllers/donations.controller.ts:746` -> Action: Add ownership checks using the authenticated user, with a separate explicit admin override where product policy allows moderation.
* Firebase sync API key fails open when `SYNC_API_KEY` is unset: the default `"change-me-in-production"` value causes `syncUser` to skip `checkApiKey`, leaving the endpoint dependent only on admin JWT. - File: `apps/api/src/controllers/sync.controller.ts:26`, `apps/api/src/controllers/sync.controller.ts:416` -> Action: Fail startup when `SYNC_API_KEY` is missing in production-like environments and always require the API key for sync endpoints.

## ЁЯРЮ Bugs & Logic Issues

* Concurrent token refresh can hang callers: when one request is refreshing and refresh fails, queued subscribers are cleared without resolution, leaving concurrent `fetchWithAuth` promises pending indefinitely. - File: `apps/mobile/auth/interceptors/authFetchInterceptor.ts:46` -> Recommendation: Notify queued subscribers with failure or reject pending promises before clearing subscriber state.
* Challenge updates use off-by-one SQL placeholders: `paramCount` is incremented after dynamic fields, then `id` and `userId` are pushed, but the `WHERE` clause uses `$paramCount` and `$paramCount + 1`, which can point to the wrong values and produce false "Challenge not found" failures. - File: `apps/api/src/controllers/challenges.controller.ts:281` -> Recommendation: Capture `idParamIndex` and `userIdParamIndex` immediately after pushing those values, then use those explicit indexes in the query.
* Follow/unfollow actor spoofing: `UserFollowService.followUser` inserts `followData.follower_id` from the request body instead of the authenticated session user, corrupting follower counts and activity attribution. - File: `apps/api/src/services/user-follow.service.ts:19` -> Recommendation: Derive follower identity server-side and remove client-controlled follower ids from protected mutation DTOs.

## ЁЯПЧя╕П Architecture & Clean Code

* Clean Architecture boundary erosion remains significant in API controllers: large controllers combine HTTP concerns, business rules, schema/table setup, direct SQL, Redis invalidation, and notifications. This violates the target separation in `docs/SSOT/CODE_QUALITY/layers-and-boundaries.md` and is contributing to inconsistent auth/ownership enforcement. - File: `apps/api/src/controllers/posts.controller.ts`, `apps/api/src/controllers/donations.controller.ts`, `apps/api/src/controllers/challenges.controller.ts` -> Action: Move use-case logic and authorization invariants into services/application-layer units as each affected endpoint is fixed.
* Authentication behavior is fragmented on mobile: `apiService` uses `fetchWithAuth`, while other utilities maintain separate fetch/token paths, making token refresh and error behavior difficult to reason about consistently. - File: `apps/mobile/utils/apiService.ts`, `apps/mobile/utils/postsService.ts`, `apps/mobile/auth/interceptors/authFetchInterceptor.ts` -> Action: Consolidate authenticated HTTP calls behind a single client with one refresh, logging, and retry policy.
