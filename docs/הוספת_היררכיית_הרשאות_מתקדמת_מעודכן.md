# תוכנית: הוספת היררכיית הרשאות מתקדמת (מעודכן ומשופר)

## סקירה כללית

הוספת מערכת היררכיית הרשאות עם דרגות דינמיות:
- **דרגה 0**: מנהל ראשי (רק `karmacommunity2.0@gmail.com`)
- **דרגה 1**: סופר מנהלים (`navesarussi@gmail.com`, `mahalalel100@gmail.com`) - משויכים למנהל הראשי
- **דרגה 2+**: מנהלים רגילים ומתנדבים - המרחק מהמנהל הראשי קובע את הדרגה
- **משתמשים רגילים**: ללא `parent_manager_id` - דרגה אחרונה (או NULL)

**חשוב**: כל מנהל הוא גם מתנדב. הדרגה נקבעת לפי המרחק מהמנהל הראשי בהיררכיה.

## עקרונות מרכזיים

### 1. חישוב דרגה
- **הדרגה = המרחק מהמנהל הראשי** (hierarchy depth)
- מחושב דינמית לפי `parent_manager_id` chain
- נשמר ב-`hierarchy_level` ומתעדכן אוטומטית בכל שינוי ב-`parent_manager_id`
- דוגמה: מנהל ראשי (0) → סופר מנהל (1) → מנהל (2) → מנהל (3) → מתנדב (4)

### 2. Roles ו-Hierarchy
- **מנהל**: יש role `admin` + `parent_manager_id` לא NULL
- **מתנדב**: יש role `volunteer` + `parent_manager_id` לא NULL
- **כל מנהל הוא גם מתנדב**: מנהלים יכולים להיות גם עם role `volunteer` (אופציונלי)
- **משתמש רגיל**: אין `parent_manager_id` → דרגה NULL או גבוהה מאוד

### 3. עדכון אוטומטי
- כל שינוי ב-`parent_manager_id` מפעיל trigger/function שמחשב מחדש את הדרגה
- הדרגה נשמרת ב-`hierarchy_level` למטרות ביצועים
- Cache invalidation אוטומטי לכל המשתמשים הרלוונטיים

## פונקציית חישוב דרגה

### PostgreSQL Function
```sql
CREATE OR REPLACE FUNCTION calculate_hierarchy_level(user_id UUID)
RETURNS INTEGER AS $$
DECLARE
    level INTEGER := 0;
    current_parent UUID;
    root_admin_email TEXT := 'karmacommunity2.0@gmail.com';
BEGIN
    -- אם זה המנהל הראשי, דרגה 0
    IF EXISTS (SELECT 1 FROM user_profiles WHERE id = user_id AND email = root_admin_email) THEN
        RETURN 0;
    END IF;
    
    -- אם אין parent_manager_id, זה משתמש רגיל
    SELECT parent_manager_id INTO current_parent 
    FROM user_profiles 
    WHERE id = user_id;
    
    IF current_parent IS NULL THEN
        RETURN NULL; -- משתמש רגיל, אין דרגה
    END IF;
    
    -- חישוב רקורסיבי של המרחק מהמנהל הראשי
    WITH RECURSIVE hierarchy_path AS (
        -- Base case: start from user
        SELECT id, parent_manager_id, 1 as depth
        FROM user_profiles
        WHERE id = user_id
        
        UNION ALL
        
        -- Recursive: go up the chain
        SELECT u.id, u.parent_manager_id, hp.depth + 1
        FROM user_profiles u
        INNER JOIN hierarchy_path hp ON u.id = hp.parent_manager_id
        WHERE hp.depth < 100 -- Prevent infinite loops
    )
    SELECT COALESCE(MAX(depth), 0) INTO level
    FROM hierarchy_path
    WHERE EXISTS (
        SELECT 1 FROM user_profiles 
        WHERE id = hierarchy_path.id 
        AND email = root_admin_email
    );
    
    -- אם לא הגענו למנהל הראשי, זה משתמש רגיל
    IF level = 0 THEN
        RETURN NULL;
    END IF;
    
    RETURN level;
END;
$$ LANGUAGE plpgsql;
```

### Trigger לעדכון אוטומטי
```sql
CREATE OR REPLACE FUNCTION update_hierarchy_level()
RETURNS TRIGGER AS $$
DECLARE
    new_level INTEGER;
BEGIN
    -- חשב דרגה חדשה
    new_level := calculate_hierarchy_level(NEW.id);
    
    -- עדכן את הדרגה
    NEW.hierarchy_level := new_level;
    
    -- אם הדרגה השתנתה, שמור היסטוריה
    IF OLD.hierarchy_level IS DISTINCT FROM new_level THEN
        INSERT INTO user_hierarchy_history (
            user_id, changed_by, old_level, new_level,
            old_parent_manager_id, new_parent_manager_id,
            old_role, new_role
        ) VALUES (
            NEW.id, 
            NULL, -- TODO: לקבל מה-user context
            OLD.hierarchy_level, 
            new_level,
            OLD.parent_manager_id,
            NEW.parent_manager_id,
            OLD.roles,
            NEW.roles
        );
    END IF;
    
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Trigger לפני UPDATE
DROP TRIGGER IF EXISTS trigger_update_hierarchy_level ON user_profiles;
CREATE TRIGGER trigger_update_hierarchy_level
    BEFORE UPDATE OF parent_manager_id, roles ON user_profiles
    FOR EACH ROW
    EXECUTE FUNCTION update_hierarchy_level();
```

## לוגיקה מרכזית

### חישוב דרגה אוטומטי
```typescript
function calculateHierarchyLevel(user):

  if email in ['navesarussi@gmail.com', 'mahalalel100@gmail.com']: 
    return 1
  
  // דרגה 2+: חישוב לפי parent_manager_id chain
  if parent_manager_id:
    level = calculateDepthFromRoot(parent_manager_id) + 1
    return level
  
  // משתמש רגיל
```

### כללי שינוי דרגה

1. **משתמש → מתנדב**: 
   - כל מנהל (דרגה 1+) יכול
   - דורש `parent_manager_id` + role `volunteer`
   - הדרגה מחושבת אוטומטית

2. **מתנדב → מנהל**: 
   - רק סופר מנהלים (דרגה 1) יכולים
   - מוסיף role `admin` (שומר גם `volunteer` אם היה)
   - הדרגה נשארת (או משתנה אם משנים `parent_manager_id`)

3. **הסרת parent_manager_id**: 
   - אוטומטית מחזיר למשתמש רגיל
   - מסיר roles `admin` ו-`volunteer`
   - `hierarchy_level` = NULL

4. **שינוי parent_manager_id**: 
   - מחשב מחדש את הדרגה אוטומטית
   - שומר היסטוריה

5. **היסטוריה**: 
   - כל שינוי נשמר ב-`user_hierarchy_history`
   - כולל: דרגה, roles, parent_manager_id

## Cache Invalidation Strategy

### Redis Cache Keys
- `user_profile_{userId}` - פרופיל משתמש
- `users_list_*` - רשימות משתמשים
- `hierarchy_tree_*` - עצי היררכיה

### Invalidation Points
1. **בשינוי parent_manager_id**:
   - המשתמש עצמו
   - המנהל הישן (אם היה)
   - המנהל החדש
   - כל הכפופים (subordinates)
   - כל המנהלים מעליו (ancestors)
   - רשימות משתמשים

2. **בשינוי roles**:
   - המשתמש עצמו
   - רשימות משתמשים

3. **בחישוב דרגה**:
   - רק אם הדרגה השתנתה

### Implementation
```typescript
async invalidateHierarchyCache(userId: string) {
  // המשתמש עצמו
  await this.redisCache.delete(`user_profile_${userId}`);
  
  // כל הכפופים
  const subordinates = await this.getSubordinates(userId);
  for (const sub of subordinates) {
    await this.redisCache.delete(`user_profile_${sub.id}`);
  }
  
  // כל המנהלים מעליו
  const ancestors = await this.getAncestors(userId);
  for (const anc of ancestors) {
    await this.redisCache.delete(`user_profile_${anc.id}`);
  }
  
  // רשימות משתמשים
  await this.redisCache.invalidatePattern('users_list_*');
  await this.redisCache.invalidatePattern('hierarchy_tree_*');
}
```

## בדיקות נדרשות

1. **בדיקת היררכיה**:
   - אין cycles
   - כל משתמש מחובר למנהל הראשי (או NULL)
   - הדרגות נכונות

2. **בדיקת הרשאות**:
   - כל דרגה יכולה רק מה שמותר לה
   - Guards עובדים נכון

3. **בדיקת migration**:
   - כל המשתמשים הקיימים מקבלים דרגה נכונה
   - דרגה 0: רק karmacommunity2.0@gmail.com
   - דרגה 1: רק navesarussi@gmail.com, mahalalel100@gmail.com
   - שאר המשתמשים: hierarchy_level = NULL

4. **בדיקת היסטוריה**:
   - כל שינוי נשמר
   - הנתונים נכונים

5. **בדיקת UI**:
   - הטאבים והפילטרים עובדים נכון
   - הדרגות מוצגות נכון

6. **בדיקת ביצועים**:
   - חישוב דרגה מהיר
   - Cache invalidation יעיל
   - אין N+1 queries

## הערות חשובות

1. **ביצועים**: הדרגה נשמרת ב-DB למטרות ביצועים, אבל מחושבת גם דינמית לוודאות
2. **Cache**: חשוב לבטל cache נכון כדי למנוע נתונים ישנים
3. **היסטוריה**: כל שינוי נשמר למטרות audit
4. **Backward Compatibility**: המשתמשים הקיימים יקבלו hierarchy_level = NULL (משתמשים רגילים)
5. **Security**: Guards חייבים לבדוק גם roles וגם hierarchy_level
